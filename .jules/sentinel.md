## 2025-05-21 - Insecure Random Number Generation for OTP and Information Leakage in Exceptions
**Vulnerability:** 1. OTP codes were generated using `random.randint` which is not cryptographically secure, making OTPs potentially predictable. 2. `str(e)` was passed directly to the `detail` of HTTP exceptions in several modules (`ai/router.py`, `payments/services.py`), exposing internal python error traces to end users.
**Learning:** 1. Python's standard `random` module uses the Mersenne Twister PRNG which is completely predictable. This is a common pattern when developers need "a random number" but do not consider the cryptographic security implications for sensitive tokens. 2. A catch-all `except Exception as e:` block can hide many specific exceptions. When `str(e)` is returned via HTTP, it may include sensitive internal logic paths, missing library data, or database-specific failures.
**Prevention:** 1. ALWAYS use the `secrets` module (e.g., `secrets.choice`, `secrets.randbelow`) or `os.urandom` for generating passwords, session tokens, security codes, or OTPs. 2. Fail securely. Log `e` internally using a logger, but return a generic, non-informative error message (like "Error confirming payment" or "Internal error occurred") to the user.
